This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is (and is not)

This repo is the **public landing site + release distribution channel** for FreeSystemDoctor, a Windows optimizer app. It is **not** the application source code — the app is a Python project (Python 3.10+, `requirements.txt` referenced in `index.html`) that lives elsewhere and is shipped here only as a prebuilt binary.

What is in this repo:

- A static, multi-page site served via GitHub Pages (`.nojekyll` is present, no build step).
- The compiled binary at `dist/FreeSystemDoctor.exe` (~25 MB, x64). Pushing a new EXE here is what publishes a release.
- Two GitHub Actions workflows that fully automate releases and the changelog.

When users say "update the app", "add a feature", or reference modules like `registry_cleaner.py` or `gui/theme.py`, those live in the upstream Python project — not here. The HTML may reference them, but the files won't exist locally.

## Build / test / lint

There is no build, test, or lint step. The site is plain HTML/CSS + a small amount of vanilla JS in `<script>` tags; open the files directly or serve the directory (`python3 -m http.server`) to preview. Don't introduce a bundler, framework, or package.json — single-file HTML is a deliberate constraint.

The EXE is built externally and committed; do not attempt to rebuild it here.

## Architecture

Three user-facing pages, one shared stylesheet, one data file, two workflows:

- `index.html` — landing page. Hard-codes the headline numbers ("62 modules", "45 tools", "Wersja 2.1", "25 MB"). When the app version changes, these strings need manual updates here — nothing else propagates them.
- `aktualizacje.html` ("Updates") — client-side page that fetches live data from `api.github.com` at load time: latest commit touching `dist/FreeSystemDoctor.exe`, file size via HEAD on the raw EXE URL, and the releases list. Falls back to commit history if no releases exist. Has its own mini Markdown renderer (`renderBody`) that only handles `###`, `- `, `---`, `**bold**`, and `` `code` `` — release-notes Markdown must stay within that subset to render correctly.
- `changelog.html` — renders `changelog.json` (fetched with cache-buster), grouping commits by date and tagging each via `classifyCommit()`.
- `changelog.json` — generated by the workflow below, not edited by hand.
- `style.css` — single shared stylesheet. The header comment notes its palette mirrors `gui/theme.py` of the upstream app; keep colors in sync if the app theme changes.

The repo slug `caytec/freesystemdoctor` is hardcoded in several JS strings (`aktualizacje.html`, `changelog.html`) and HTML links (`index.html` footer/nav). If the repo is ever renamed or forked, grep for it.

### The two workflows (this is the core of the repo's automation)

`.github/workflows/release-on-exe.yml`
- Triggers **only** on pushes to `main`/`master` that change `dist/FreeSystemDoctor.exe` (path filter), or manual dispatch.
- Tags as `v{YYYY.MM.DD}-{short-sha}` and uses `softprops/action-gh-release@v2` with `make_latest: true`.
- An inline Python script generates release notes by `git log`-ing commits since the previous tag, classifying each via regex into five Polish-labeled buckets (`feat`/`fix`/`refactor`/`docs`/`chore`), stripping the prefix, and appending a footer with build metadata. Commits containing `[skip ci]` are filtered out.

`.github/workflows/update-changelog.yml`
- Triggers on every push to `main`/`master` **except** when only `changelog.json` changed (paths-ignore), to avoid loops.
- Regenerates `changelog.json` (last 500 commits, no merges) and commits it back as `chore: auto-update changelog [skip ci]`.
- The `[skip ci]` marker is what prevents the release workflow from picking up bot commits.

Implication: never hand-edit `changelog.json` (it'll be overwritten on the next push) and never remove the `[skip ci]` from the bot's commit messages (it'll cause loops and noise in release notes).

## Commit message conventions

Both the release-notes generator and the in-page `classifyCommit()` use the same regex set, so commit prefixes drive what users see. Polish prefixes are first-class alongside English:

- `feat` — also matches `add`, `nowy`, `dodaj`, `stwórz`, `nowe`, `new`
- `fix` — also matches `napraw`, `popraw`, `bug`
- `refactor` — also matches `refaktor`, `cleanup`, `przepisz`
- `docs` — also matches `doc`, `readme`
- `chore` — default bucket; also matches `build`, `ci`, `deps`, `bump` (in the JS classifier)

Anything not matching falls into `chore`. The prefix is stripped from the displayed message, so write commit subjects that read naturally after the prefix is removed.

Append `[skip ci]` to any commit you don't want surfaced in release notes or the changelog.

## Language / locale

All user-facing copy is Polish (`<html lang="pl">`, dates formatted with `pl-PL` locale). When editing copy, keep it Polish — don't translate to English. Code identifiers, commit prefixes, and filenames stay English.

## Releasing a new app version

The flow is: update `dist/FreeSystemDoctor.exe` → push to `main` → release workflow fires automatically. If the version number in the binary changed, also update the hard-coded strings in `index.html` (hero badge "Wersja X", download card "v X", file-size badges) — there is no template variable for these.
